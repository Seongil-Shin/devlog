## 모노레포 탄생 배경


- 모놀리식
	- 모듈화 설계된 소프트웨어 애플리케이션
	- 장점 : 아키텍처를 빠르게 구축할 수 있음
	- 단점 : 유기적으로 코드가 연결되어 관심사의 분리가 어려워짐
- 모듈식
	- 애플리케이션을 모듈로 분리한 방식.

만약 애플리케이션에서 각 모듈들을 다른 애플리케이션에서도 사용해야한다면 어떻게 해야할까? 이때는 모듈을 별로 레포지토리로 분리하고 독립적으로 빌드가능하게 하여 다른 애플리케이션에서도 사용가능하게 할 수 있다. 이를 **멀티레포 방식**이라고 하고, 모듈별로 별도 레포지토리를 갖기에 자유도가 높다.

하지만 멀티레포에는 단점이 있다
- 관리해야하는 프로젝트가 늘어가게 되면 개발환경이나 빌드, 테스트, 배포 파이프라인 등을 반복해서 구성해야함
- 서드파티 라이브러리 버전 의존에 따른 호환성 이슈가 발생 (모듈별로 다른 라이브러리 사용할 시 이슈 발생)
- 중복 코드 발생
- 일관되지 않은 코드 발생

이러한 멀티레포의 단점을 해결하기위해 모노레포 방식이 나왔다


## 모노레포란?

모노레포란 하나의 레포지토리에서 다수의 프로젝트를 관리하는 방식이다. 하나의 레포지토리를 사용하기에 의존성 관리가 더 쉽고, 관리포인트 또한 단일화가 되었다.

장점
- 여러 프로젝트간 코드의 공유와 재사용이 쉬움
- 모노레포내 프로젝트 개발자간 협업이 쉽다. (일관된 도구, 동일한 컨벤션)
- 통합된 개발환경과 devops 구성
- 새프로젝트를 쉽게 만들 수 있음
단점
- 의존성이 연결되어 과도한 의존관계가 나타날 수 있다.
- 하나의 CI를 구성하기 어려운 경우도 있다
- 모든 코드가 밀집되어있기에 작은 문제가 커질 수 있다

## 모노레포의 구성

모노레포를 구성하는 방식은 크게 두가지로 나뉜다

### 모노레포 빌드 시스템 도구 없이 package manger로 구성

- npm, yarn, pnpm 모두 workspace 기능을 통해 모노레포를 지원한다
- workspace는 하위 프로젝트를 구성하고 자체적인 종속성 트리를 구성하게 한다. 이를 통해 하나의 레포 아래에서 분리된 프로젝트를 만들 수 있게 된다.
- 또한 package manager의 workspace 관련 명령어를 통해 workspace 간 종속성을 효율적으로 관리할 수 있도록 도와준다.
- package manager 만으로 구성하게 되면 모노레포 빌드 시스템 도구를 사용하여 나타나는 오버헤드를 줄일 수 있게 된다.
- 다만 프로젝트 규모가 커지면 빌드 타임을 줄이기 위해 필요한 캐싱, 분산 작업과 같은 구성이 어려워질 수 있다

### 모노레포 빌드 시스템 도구를 같이 사용하는 방식

- 빌드 시스템 도구를 사용하여 workspace가 점차 성장함에따라 늘어나는 관리포인트를 쉽게 해결해줄 수 있다
- 변경점에 영향을 받는 프로젝트 빌드 테스트만 실행할 수 있다
- 별다른 구성없이 workspace 또는 프로젝트 간의 의존성 그래프를 쉽게 확인할 수 있게 해준다.


## 모노레포 with pnpm 

장점
- npm 과 사용법이 비슷
- package.json에 peer dependency를 명시하지 않으면 사용할 수 없는 엄격함 -> 실수 방지
- 설치,실행이 빠름
- 용량 절약
- 모듈 추가가 빠름
단점
- 낯섬
- 국내 사용량이 적음
