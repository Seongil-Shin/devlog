## 카프카 개요 및 설명

처음에는 source application과 target application 과의 통신이 단방향이었지만 시간이 갈수록 다양한 요소가 생기면서 복잡해졌다.

아파치 카프카는 source 와 target 사이의 커플링을 약하게 하기 위해 만들어졌다.
- source(producer)는 데이터를 보내기만 한다
- target(consumer)은 데이터를 받기만 한다

카프카 특징
- 고가용성 : 서버에 이슈가 생기는 상황에서도 손실없이 복구가능함
- 낮은 지연과 높은 처리량으로 빅데이터 처리에 유용하다

## 카프카 토픽

토픽 = 데이터베이스의 테이블, 파일시스템의 폴더와 유사함.
- Producer가 토픽에 데이터를 넣으면 Consumer가 가져갈 수 있음. 
- 토픽에는 이름을 지정할 수 있다.

토픽의 내부 구조
- 하나의 토픽은 여러개의 파티션으로 구성됨
- 데이터는 파티션에 큐와같이 차곡차곡 쌓이게 된다. Consumer는 이 데이터를 차례대로 받을 수 있다
- 만약에 데이터가 어느정도 쌓인 상태에서 새로운 Consumer가 연결되었을 경우 이전에 쌓여진 데이터를 받을 수도 있다.
	- 컨슈머 그룹이 다르며, `auto.offset.reset = earliest` 설정이 되어있을 경우
- 데이터는 Consumer가 가져가도 삭제 되지 않으며, 삭제되는 타이밍은 옵션에 따라 지정할 수 있다
- 파티션이 두개 이상인 경우
	- 키가 null이고 기본 파티셔너 사용중인 경우 : round-robin으로 데이터가 들어간다
	- 키가 있고, 기본 파티셔너 사용중인 경우 : 키의 해시 값을 구해서 특정 파티션에 들어간다
- 파티션을 늘리는 것은 가능하지만, 파티션을 줄일 수는 없다.
	- 파티션을 늘리면 컨슈머의 개수를 늘려서 데이터 처리를 분산시킬 수도 있다


## Broker, Replication, ISR


Broker
- 카프카가 설치되어 있는 서버 단위
- 보통 3개를 권장한다

Replication
- 파티션의 복제를 뜻하며, 최대 브로커의 개수만큼 존재할 수 있다.
- Replication 중 한대를 원본(Leader)라고 하며 데이터를 받는 곳으로 지정한다. 나머지는 복제본(Follwer)들이며, 원본에서 받은 데이터를 복사해서 저장한다.
- 파티션의 고가용성을 위해 사용함
- 리더 브로커에 문제가 생기면 팔로워가 리더를 승계할 수 있다
- Replication은 너무 많아지면 리소스 사용량이 많아지기에 3개 이상의 브로커에서도 3개 정도를 권장한다

ISR
- In Sync Replica 의 약자로, Leader + Follwer 파티션을 합쳐서 부르는 말.


ACK
- Producer 의 옵션으로 옵션에 따라 대응가능한 가용성 범위가 달라진다
- 0일 경우 : producer가 전송 후 응답 값을 받지 않음. 따라서 빠르지만 데이터 유실 가능성 있음
- 1일 경우 : 전송 후 leader가 제대로 받았다는 응답을 받음. 하지만 복제가 완료되었는지는 받지 않음
- all일 경우 : 전송 후 복제까지 완료되었다는 응답을 받음. 데이터 유실은 안되지만, 속도는 느리다.
