## 브라우저는 서버를 신뢰하지 않는다

아래와 같은 html 코드는 우선순위에 맞게 로드가 되어야한다. 브라우저는 이를 순서대로 priority에 맞게 순서대로 요청한다.

![](assets/images/Pasted%20image%2020250123153355.png)

이때, HTTP/2, HTTP/3 는 멀티플렉싱을 제공하여 하나의 연결에 여러 요청과 응답을 주고 받을 수 있는데, 서버에서 이를 잘못 처리하면 다음과 같이 브라우저가 요청한 순서대로 응답을 내보내는게 아닌 뒤죽박죽 보내게 된다.

![](assets/images/Pasted%20image%2020250123153542.png)

network bandwidth는 제한이 있으므로 이렇게 뒤죽박죽 보내게 되면 결과적으로 빠르게 받아야할 리소스를 늦게 받게 되는 결과가 발생하게 된다.

이러한 이유로 브라우저는 서버를 신뢰하지 않고 리소스 별로 우선순위를 두어 한 덩어리의 요청이 종료되었을때 다음 우선순위를 요청하는 식으로 최적화를 진행하였다. 이러한 최적화는 브라우저마다 다르게 적용되어있다.

## 크롬의 경우

크롬은 `Tight mode`라는 것을 가지고 있어 low priority resource의 로드를 제한한다. (2개 이하이면 `tight mode`에서도 로드됨)

> Chrome loads resources in 2 phases. “Tight mode” is the initial phase and constrains loading lower-priority resources until the body is attached to the document (essentially, after all blocking scripts in the head have been executed). In tight mode, low priority resources are only loaded if there are less than 2 in-flight requests at the time that they are discovered.
> [공식문서](https://docs.google.com/document/d/1bCDuq9H1ih9iNjgzyAL0gpwNFiEP4TZS-YLRp_RuMlc/edit?tab=t.0)

이때 크롬에서 구분하는 리소스별 우선순위와 Tight mode 에서 로드되는 우선순위는 다음과 같다

![](assets/images/Pasted%20image%2020250123154359.png)

따라서 2개의 high js와 10개의 low img가 있는 페이지는 다음과 같이 로드된다.
![](assets/images/Pasted%20image%2020250123154526.png)

하지만 크롬 117 부터는 medium-priority img 도 tight mode에서 2개씩 로드 된다. 

> As of Chrome 117, Chrome will also load 2 Medium-priority requests at a time with no restrictions about other requests being in-flight.

![](assets/images/Pasted%20image%2020250123154607.png)

따라서 117 부터는 아래와 같이 로드된다.

![](assets/images/Pasted%20image%2020250123154648.png)

여기서 만약 high js 중 하나가 굉장히 빨리 로드되면 어떻게 될까? 공식 문서에는 아래와 같이 설명되어있다.

>In tight mode, low priority resources are only loaded if there are less than 2 in-flight requests at the time that they are discovered.

이처럼 만약 진행중인 요청이 2개 이하이면 2개를 채울때까지는 low img 도 로드된다.

![](assets/images/Pasted%20image%2020250123154912.png)

또 다른 예시로 만약 High JS가 없다면 Tight mode는 아래와 같이 동작한다. blocking scripts가 없기에 low resource가 바로 로드된다.

>essentially, after all blocking scripts in the head have been executed

![](assets/images/Pasted%20image%2020250123155037.png)


## 사파리

사파리도 크롬과 비슷하게 tight mode를 가지지만 아래와 같은 공통점과 차이점이 존재한다
- 공통점
	- tight mode로 불러오는 요청이 2개 이하일 경우 low img 요청이 간다는 것은 크롬과 동일하다
	- body의 밑에서 불러오는 js는 tight mode를 발생시키지 않는다
- 차이점
	- 첫 5개 이미지에 대한 최적화는 진행하지 않는다
	- css 도 tight mode를 발생시킨다.
	- head 내부에 있는 `async`, `defer` js 를 tight mode에서 로드 시킨다
	- body 중간에서 불러오는 js의 경우 tight mode를 발생시킨다. 하지만 다소 다르게 발생시킨다
![](assets/images/Pasted%20image%2020250123163954.png)


## 파이어폭스

HTTP/2, HTTP/3 에 대해서 tight mode를 가지고 있지 않는다.


## Tight mode를 조절하는 방법

Tight mode 에 대해서는 배웠으니 low resource를 어떻게 tight mode에서 로드할 수 있을지, tight mode 에서 로드되는 리소스를 어떻게 제외할 수 있는지 알아보자

### low resource -> tight mode

- `fetchpriority=high`
	- chrome : images, defet/async js, body의 밑에 있는 js 의 경우 tight mode에서 로드할 수 있도록 해줌
	- safari : images 에 대해서만 tight mode에서 로드되도록 해줌
- `<link rel="preload" src="lcp.jpg">`
	- chrome : 동작안함
	- safari : 처음 2개만 tight mode에서 로드되도록 해줌. (아무리 많이 달아도 2개만 적용됨)
- `fetchpriority=high` + `<link rel="preload" src="lcp.jpg">`
	- chrome, safari : 모두 잘 tight mode에서 동작함

다만 이미지의 경우 preload가 없어도 tight mode에서 잘 로드되니 html에 있다면 필요하지 않다. 엣지케이스나 동적으로 불러오는 이미지(js, css 등에서)에 대해서만 preload가 필요하다.

### tight mode -> low
- `fetchpriority=low`
	- chrome : first 5 images, js/css late in body, preloaded fonts, preloaded async/defer js 
	- safari : 동작안함



## 성능 이슈에 대해서

Tight mode에 대해 잘 알아봤지만 그렇게 큰 영향을 끼치지는 않는다. CDN 등 네트워크 구조로 문제를 해결하는 것이 더 큰 영향을 끼친다.
