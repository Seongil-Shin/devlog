## CSRF 란?

사용자가 자신의 의지와 무관하게 공격자가 의도한 행동을 하게 만드는 공격.

공격할 사이트를 분석하여 URL에 있는 토큰이나 패턴을 분석하여 의도치 않는 행동을 하게 할 수 있다

1. 사용자가 보안이 취약한 서버에 로그인한다
2. 사용자 인증 정보가 사용자의 브라우저 쿠키에 저장된다
3. 공격자는 사용자가 악성 스크립트 페이지를 누르도록 유도한다
	- 게시판에 악성 스크립트를 삽입
	- 메일 등으로 악성 스크립트 전송 
4. 사용자가 악성 스크립트가 작성된 페이지 접근 시 브라우저의 쿠키를 사용하여 의도치 않은 요청을 서버로 보낸다.


ex)
- 메일로 "http://example.com/user.do?cmd=user_passwd_change&user=admin&newPwd=1234" 라는 주소를 보내어 사용자의 비밀번호를 바꾸는 방법 (사용자가 해당 사이트에 로그인하여 쿠키가 남아있는 상태)
- 이미지 태그에 비슷한 주소를 넣어 요청을 하게 만드는 방법


## 예방법

Referer 체크 추가
- 요청을 보낸 사이트가 어딘지 헤더에 담고있는 정보. 해당 사이트 내에서 온 요청이 아니면 거절한다
- 헤더를 조작할 수 있기에 완전한 방법은 아니다.

CSRF 토큰 발급
- 세션에 임의의 토큰을 발급하여 해당 토큰 없이 요청할 경우 요청을 거절하는 방법

CAPTCHA 사용
- CAPTCHA 인증 코드가 없을 경우 요청을 거부하는 방법

GET/POST 구분
- 메일로 공격 주소를 보내는 경우, 이미지 태그에 넣는 경우 GET으로 요청이 가기에 POST로만 요청을 받도록 하여 어느정도 막을 수 있다

Cookie SameSite 설정
- SameSite 설정을 통해 어느정도 막을 수 있다
- Strict : public suffix(com, net)을 기준으로 한단계 하위 도메인까지만 확인해서 같은 도메인 (first-party context)일 경우에만 쿠키 전송
- LAX : strict 보다는 느슨한 정책. Top Level Navigation(`<a>`, `window.location.replace`, 302 등으로 이동하는 것)이거나 GET과 같은 안전한 HTTP 메서드 요청일 경우 전송됨.
- None : 모든 상황에서 함께 전송됨. secure 플래그를 사용해야한다


## vs XSS

XSS는 웹사이트에 의도치 않은 스크립트를 넣어서 실행시키는 기법이다.

공격자의 입력 값을 제대로 처리하지 않고 사용할 경우 발생하며, 다른 사용자가 공격자의 스크립트를 볼 경우 쿠키, 세션 등의 정보를 탈취당할 수 있다.

XSS는 사용자가 특정 사이트를 신뢰하기에 발생하는 것이라면 CSRF는 특정 사이트가 사용자를 신뢰하기에 발생하는 문제라 볼 수 있다.

예방법은 다음과 같다
- 중요정보는 쿠키 대신 서버에 저장한다
- 정보를 암호화한다
- cookie에 httpOnly 옵션을 설정하여 cookie를 빼내지 못하도록 한다
- 공격자의 입력을 검사한다
